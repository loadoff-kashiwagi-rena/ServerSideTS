# 調査・比較ドキュメント生成

技術選定や方式比較を、このリポジトリの既存ドキュメント（`技術検証.md` 等）と同じ型で書く。
読者が「なぜこの方式を選んだか」を後から追えることをゴールにする。

## 構成（この順で書く）

```markdown
#### <比較テーマ>（<方式A ／ 方式B>）

<1〜3文で前提と背景。なぜ比較が必要か。併用可否などの制約があれば明記。>
現状は **<採用中の方式>** を採用。

- 現状：<方式Aの経路や仕組みを一行で（矢印図など）>
- 比較対象：<方式Bの経路や仕組みを一行で>

| 観点 | <方式A> | <方式B> |
|---|---|---|
| <観点1> | … | … |
| <観点2> | … | … |

**評価**：

<どの前提・要件のもとで、どちらを選ぶか／保留するかを言い切る。>
<判断が条件付きなら「○○が確定したら再判断」と保留条件を明記する。>

参考：<参照したURL（あれば）>
```

## 観点の選び方

比較表の「観点」は、テーマに応じてこの中から関連するものを選ぶ（不足は補う）:
セキュリティ／保護経路、IP・アクセス制限の実現方法、構成のシンプルさ、
コスト、可用性・スケーラビリティ、運用負荷、有効期限・認証の性質、将来の拡張性。

各セルは「⭕/✅ 利点」「⚠️/❌ 欠点・注意」を絵文字で明示すると既存トーンに合う。

## 手順

1. 比較対象の各方式について、会話・コード・既存 `.md`（`技術検証.md` など）から事実を集める。曖昧な点は推測せず「確認中」とする。
2. 上の構成で書く。比較表は観点ごとに両方式を対置する。
3. **評価**は必ず「現在の要件・前提」を明記したうえで結論を出す。要件が未確定で判断保留なら、何が確定したら決められるかを書く。
4. 出力先：既存の検討ドキュメント（`lambda-express-api/技術検証.md` 等）の該当セクションに `####` 見出しで追記するか、独立トピックなら新規 `.md` を作る。ユーザーに出力先を確認する。

## 注意

- 一方を不当に有利に書かない。両方式の欠点を必ず1つ以上挙げる。
- 「併用不可」「制約」など判断を左右する前提は冒頭で明記する。
- コスト・性能などの数値が未確定なら断定せず、判断保留の条件として残す。
