#### ネットワーク / 接続要件
- S3 バケットは **バリデーション前S3 / バリデーション後S3 を別バケットとして分離**（IAM・バケットポリシー・ライフサイクルを個別管理）
- VPC 内 Lambda から各サービスへの到達経路を確保する（RDS をプライベートサブネットに隔離する前提）
  - **S3 Gateway VPCエンドポイント**：VPC内 Lambda から S3（バリデーション前 / バリデーション後S3バケット）へアクセスするため（無料）
  - **Secrets Manager Interface VPCエンドポイント**（or NAT Gateway）：`getPool()` が DB認証情報を Secrets Manager から取得するため
- **RDS Proxy**：Lambda 同時実行による RDS コネクション枯渇対策（コネクションプール集約）。`RDS_PROXY_SETUP.md` 参照

#### 【技術検証】Lambda から RDS への接続はどうなるか（ローカル実測）

##### まず用語（初学者向け）
- **接続（コネクション）**：アプリと DB をつなぐ「電話回線」のようなもの。1本つなぐと、切るまで回線を1本占有する。
- **コネクションプール**：毎回つなぎ直すのは遅いので、開いた回線を**何本かまとめて持っておいて使い回す**仕組み。今回は最大10本（既定）。
- **Lambda のウォーム/コールド**：Lambda は一度動くと、しばらく「温かい状態（ウォーム）」でプロセスが残る。次の呼び出しはそのプロセスを再利用するため、**前回開いた回線（プール）もそのまま残っている**。完全に止まって作り直すのがコールド。

##### 何を確かめたか
「Lambda から DB に直接つなぐと、回線はいつまで残るのか？ たくさん同時に来たら回線は足りるのか？」をローカルの MySQL で実測した。

##### 実測結果
| 確かめたこと | 結果 | かみ砕くと |
|---|---|---|
| DB がアイドル回線を自動で切る時間（`wait_timeout`） | 28800 秒（8時間） | DB 側は8時間放置されないと回線を切らない＝ほぼ切らない |
| DB がさばける回線の上限（`max_connections`） | 151 本 | この DB は同時に151本までしかつなげない |
| クエリ1回で開く回線 | 1 本 | 使った分だけ開く（最初から10本ではない） |
| 同時に20回クエリを投げた後 | **10 本**（プール上限まで増加） | 混むと一気に上限の10本まで開く |
| そのまま75秒放置 | **10本のまま減らない** | 暇になっても回線を手放さない（既定設定では刈り取られない） |

##### ここからわかること（なぜ RDS Proxy が要るか）
- **Lambda 1個（1プロセス）だけで、最大10本の回線を握りっぱなしにする。**
- Lambda はアクセスが増えると自動でプロセスを増やす（同時実行）。すると **「プロセス数 × 10本」** で回線が増える。
- DB の上限は151本なので、**Lambda が15個も同時に動けば回線が枯渇**し、新しいリクエストは「DB につなげない」エラーになる。本番 DB の上限はもっと小さいことが多く、さらに少ない数で枯渇する。
- **RDS Proxy = 電話の交換手**。多数の Lambda からの回線要求を、交換手が**少数の DB 回線にまとめて取り次ぐ**。これで「プロセス数 × 10本」の増え方を抑え、枯渇を防ぐ。だから本構成では RDS Proxy を挟む。

> 補足：上記はローカル MySQL での確認。実際の Lambda の「ウォーム再利用」や「RDS Proxy の効果」は AWS 上で、CloudWatch の `DatabaseConnections`（DB 側接続数）が同時実行に比例して増えるか／RDS Proxy で頭打ちになるか、を見て確認する。

### 情報処理
![presignedUrl](image/presignedUrl.png)

保存ファイルにMP4があるため、データ量が大きい。
API Gateway の 10MB・Lambda の 6MB（同期呼び出しのリクエスト/レスポンス上限）に引っかかるため、「Presigned URL」の使用。
バリデーション前S3に保存（バリデーション前S3とバリデーション後S3は別バケットとして分離）。

![prodS3](image/prodS3.png)

S3への保存が完了したら、クライアントが完了通知API（`POST /uploads/complete`）を呼び出すことで Lambda を起動。
Lambda内の「バリデーション」によりバリデーション前S3に保存された内容を確認。
チェックが通ったらバリデーション後S3に保存。
通らなかったら、バリデーション前S3から削除し、レスポンス（400 等）をブラウザに返す。

**バリデーション仕様**（バリデーション前S3 取得後・サーバー側で実施）
- 共通
  - 最大サイズ（種別ごとに上限値を定義。データ量未確定のため具体値は確認中）
  - 拡張子ホワイトリスト（クライアント申告値・補助的）
  - Content-Type ホワイトリスト（クライアント申告値・補助的）
- バイナリ（mp4 / png / jpeg）
  - マジックナンバー検証（＝先頭バイトのシグネチャ。申告 MIME と一致するか）
- テキスト（css / HTML）
  - マジックナンバー無し → サイズ＋拡張子
- ※ Content-Type・拡張子はクライアント申告値のため信用しない。最終判定はサーバー側のバイナリ検証で行う。

![response](image/RDS.png)

バリデーション後S3へ保存したら、そのS3 URL（キー）を RDS Proxy 経由で RDS に保存（RDS Proxy がコネクションプールを集約し、Lambda 同時実行による接続枯渇を防ぐ）

**整合性（S3 と RDS の分散更新）**

S3 と RDS はまたいだトランザクションにできないため、部分失敗に備えて補償処理を行う。
- バリデーション後S3へコピー後に RDS INSERT が失敗した場合 → バリデーション後S3のコピーを削除（補償）し、バリデーション前S3も削除する
- RDS 保存成功後のバリデーション前S3 削除に失敗した場合 → 処理は成功扱いとし、ログに残す（孤児ファイルとして残り得る）
- 孤児ファイル対策として、バリデーション前S3 に **ライフサイクルルール（N日で自動削除）** を設定する
- `/uploads/complete` は**冪等**に設計する（二重呼び出しでレコードが重複しないよう `s3_key` を UNIQUE 等）

#### セキュリティ方針
- IP制限（許可IP帯からのアクセスのみ。固定IP前提）
- VPC（RDS等をプライベートサブネットに隔離）
- 押せるアプリ制限（許可アプリ識別子 / APIキー）
- 認証認可は実装しない（許容するリスクとして以下を明記）
  - 許可IP帯・許可アプリの範囲内では、個人単位の権限制御・本人特定・操作監査は行わない
  - クライアントに埋め込むアプリ識別子 / APIキーは逆コンパイルで抽出され得るため、秘匿性は保証しない
  - したがって本方式は「アクセス元が固定IP（社内NW / VPN）に限定できる」前提で成立する
- **presigned URL による S3 直アップロード経路は API Gateway を通らないため、API Gateway 側の IP制限では保護されない**
  - クライアントは presigned URL で S3 へ直接 PUT するため、API Gateway / WAF の IP制限は当該経路に適用されない
  - presigned URL は有効期限内（現状5分）であれば URL 保有者なら誰でもアップロード可能な bearer 相当
  - 対策：
    - **S3 バケットポリシーで `aws:SourceIp` 条件**を付与し、S3 側でも IP制限する（採用）
    - presigned URL を **PUT ではなく POST + 条件（`content-length-range` / `starts-with`）** とし、アップロード時点でサイズ・プレフィックスを強制する
    - 有効期限は短く保つ（現状5分で妥当）

**`aws:SourceIp` によるS3側IP制限（採用方針）**

presigned URL 経由の直 PUT は API Gateway / WAF を通らないため、バリデーション前S3バケットのバケットポリシーで `aws:SourceIp` を付与し、**許可IP帯（固定IP：社内NW / VPN）以外からの S3 アクセスを拒否**する。
これにより、URL が漏えいしても許可IP外からはアップロードできない。
- 適用対象：バリデーション前S3バケット（直アップロード先）。必要に応じバリデーション後S3にも適用
- 条件キー：`aws:SourceIp`（許可CIDRのリスト）
- ポリシー例（許可IP帯以外の `s3:*` を Deny）：

```json
{
  "Sid": "DenyIfNotFromAllowedIp",
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": [
    "arn:aws:s3:::<バリデーション前S3バケット>",
    "arn:aws:s3:::<バリデーション前S3バケット>/*"
  ],
  "Condition": {
    "NotIpAddress": { "aws:SourceIp": ["203.0.113.0/24", "198.51.100.10/32"] }
  }
}
```

- 注意：`aws:SourceIp` は**送信元グローバルIP**で判定するため、許可IPは固定IP前提（要件のセキュリティ方針と整合）。VPC内 Lambda が S3 Gateway VPCエンドポイント経由で同バケットへアクセスする経路を Deny で巻き込まないよう、エンドポイント経由は `aws:SourceVpce` 等で除外する

#### アップロード方式の比較（S3 Presigned URL ／ CloudFront署名付きURL + OAC）

大容量ファイルの直アップロードには2つの方式がある。
**両者は併用不可**（S3 Presigned URL と OAC を同時に使うと `Only one auth mechanism allowed` エラー）。
現状は **S3 Presigned URL 方式** を採用。

- 現状：`[クライアント] --直PUT(presigned URL)--> [バリデーション前S3]`（**API GW を通らない**）
- 記事方式：`[クライアント] --署名付きURL--> [CloudFront] --OAC(SigV4)--> [S3]`（**全リクエストが CloudFront を通る**）

| 観点 | 現状：S3 Presigned URL | CloudFront署名付きURL + OAC |
|---|---|---|
| アップロード経路の保護 | ⚠️ API GW を通らない（上記課題）。WAF/IP制限が効かない | ✅ 全リクエストが CloudFront を通る。WAF/IP制限/DDoS対策が効く |
| IP制限の実現方法 | S3 バケットポリシーの `aws:SourceIp` で別途担保 | CloudFront + WAF で一元化 |
| S3への直アクセス遮断 | バケットポリシーで制御 | ✅ OAC で「CloudFront経由のみ」を強制 |
| 構成のシンプルさ | ✅ CloudFront 不要・部品が少ない | ⚠️ CloudFront + キーグループ + OAC が増える |
| 有効期限・bearer問題 | URL 保有者なら誰でも PUT 可（現状5分） | 署名付きURLも同性質だが、CloudFront層でWAF等を重ねられる |
| マルチパートアップロード | presigned URL + マルチパート | OAC で `READ/WRITE`、`ALL_VIEWER_EXCEPT_HOST_HEADER` が必要 |
| コスト | ✅ 安い | CloudFront 転送料が乗る |

**評価**：

上記セキュリティ方針（固定IP・社内NW/VPN前提、認証認可なし）では、S3 バケットポリシーの `aws:SourceIp` で要件を満たせるため、現状の S3 Presigned URL 方式を継続。

**CloudFront署名付きURL + OAC 方式の採否は、データ量の確認が取れてから判断する。**
- 同方式は全リクエストが CloudFront を経由するため、**転送データ量に応じて CloudFront のデータ転送料が発生**する。mp4 を含む大容量・大量配信になるとコストが無視できない。
- 一方でアップロード/配信のデータ量が小さければ、得られる追加保護（WAF/IP制限の一元化・DDoS対策）に対してコスト影響は軽微。
- したがって「保存ファイル仕様（データサイズ・ファイル形式・枚数）」と非機能要件の「アップロード最大ファイルサイズ」「想定データ件数 / 保存容量」が確定した段階で、**コスト試算 vs セキュリティ便益**を比較して採否を決める。
- それまでは S3 Presigned URL + `aws:SourceIp` 方式で要件を満たす。将来 IP制限を緩める／公開範囲を広げる場合も、API GW を通らない弱点が構造的に解消される本方式への移行を併せて検討する。

参考：https://dev.classmethod.jp/articles/cloudfront-s3-signature-oac-comparison/

### より大きいデータの場合

- presigned URL ＋　マルチパートアップロード

大きいファイルを分割してS3に保存する方法。単一の presigned PUT は1オブジェクト最大5GB。これを超える／分割再開したい場合にマルチパートを使う。

#### マルチパートアップロードの登場人物と発行物

**`UploadId` を発行するのは S3。CloudFront ではない**（CloudFront を挟む場合でも、CloudFront は署名付きリクエストを S3 に転送する「通り道」にすぎない）。

| 発行物 | 誰が作るか | 何のため |
|---|---|---|
| **UploadId** | **S3**（`CreateMultipartUpload` のレスポンス。呼ぶのは Lambda） | この分割アップロード一式を識別するID |
| 各パートのアップロード認可（署名付きURL） | **Lambda** | クライアントが各パートを PUT できるようにする |
| **ETag**（パートごと） | **S3**（各 `UploadPart` のレスポンス） | 完了時に「どのパートが正しく上がったか」を照合する |

#### フロー（S3 Presigned URL 方式）

1. クライアント → Lambda：「アップロード開始」
2. Lambda → S3 `CreateMultipartUpload` → **`UploadId` を取得**
3. Lambda → クライアントへ **`UploadId` ＋ 各パート用の presigned URL**（PartNumber と UploadId を含めて署名）を渡す
4. クライアント → 各 presigned URL に**パートを直接 PUT** → S3 が**パートごとに `ETag`** を返す
5. クライアント → Lambda へ `[{PartNumber, ETag}, ...]` を送る
6. Lambda → S3 `CompleteMultipartUpload`（UploadId ＋ パート一覧）で結合

→ クライアントが持つ必要があるのは **`UploadId`（全パート共通）** ／ **各パートを上げる署名付きURL** ／ 完了時に返す **`PartNumber + ETag` のリスト**。

```mermaid
sequenceDiagram
    autonumber
    participant C as クライアント
    participant L as Lambda
    participant S as S3（バリデーション前）

    C->>L: アップロード開始（ファイル名・分割数など）
    L->>S: CreateMultipartUpload
    S-->>L: UploadId
    L-->>C: UploadId ＋ 各パート用 presigned URL

    loop パートごと（PartNumber 1..N）
        C->>S: PUT パート（presigned URL へ直接）
        S-->>C: ETag（このパートの識別子）
    end

    C->>L: 完了通知 [{PartNumber, ETag}, ...]
    L->>S: CompleteMultipartUpload（UploadId ＋ パート一覧）
    S-->>L: 結合完了（オブジェクト確定）
    L-->>C: アップロード完了レスポンス
```

#### CloudFront + OAC 方式の場合の違い
- 流れは上とほぼ同じだが、クライアントは S3 ではなく **CloudFront ドメイン**に各リクエスト（UploadPart 等）を投げ、CloudFront が OAC(SigV4) で署名して S3 に転送する。
- それでも **`UploadId` は S3 が発行**し、Lambda 経由でクライアントに渡る（CloudFront は発行しない）。
- 注意：マルチパートはクエリ文字列（`uploads` / `uploadId` / `partNumber`）を使うため、**CloudFront がそれらを S3 へ転送し SigV4 署名に含める設定**が必要（OAC の `READ/WRITE`、オリジンリクエストポリシー `ALL_VIEWER_EXCEPT_HOST_HEADER`）。
- 補足：1回の `CopyObject`／`UploadPartCopy` でない通常コピーは5GB上限のため、バリデーション後S3へ移送側で5GB超を扱うならコピーもマルチパート対応が必要。

#### バリデーションフロー（アップロード完了 → 検証 → バリデーション後S3へ保存）

クライアントがバリデーション前S3へアップロードした後、`POST /uploads/complete` を起点に検証・バリデーション後S3へ移送・RDS保存を行う流れ（`index.ts` の実装に対応）。

**成功フロー（200系：201 Created）**

```mermaid
sequenceDiagram
    autonumber
    participant C as クライアント
    participant L as Lambda
    participant TS as バリデーション前S3（temp/）
    participant PS as バリデーション後S3（uploads/）
    participant P as RDS Proxy
    participant DB as RDS

    Note over C,TS: 事前：presign で発行した URL でバリデーション前S3 に直接アップロード済み

    C->>L: POST /uploads/complete（key, user_id）
    L->>TS: HeadObject（存在・サイズ・Content-Type 取得）
    TS-->>L: メタデータ（ContentLength / ContentType）
    Note over L: バリデーション OK（サイズ・Content-Type・マジックナンバー）
    L->>PS: CopyObject（バリデーション前 → バリデーション後S3へ移送）
    L->>P: INSERT（user_id, filename, s3_key, file_size）
    P->>DB: クエリ実行（接続を多重化）
    DB-->>P: OK
    P-->>L: OK
    L->>TS: DeleteObject（バリデーション前を削除・失敗しても成功扱い）
    L-->>C: 201（保存成功・バリデーション後S3のキーを返す）
```

**エラーフロー（4xx / 5xx）**

```mermaid
sequenceDiagram
    autonumber
    participant C as クライアント
    participant L as Lambda
    participant TS as バリデーション前S3（temp/）
    participant PS as バリデーション後S3（uploads/）
    participant P as RDS Proxy
    participant DB as RDS

    C->>L: POST /uploads/complete（key, user_id）
    L->>TS: HeadObject（存在確認・メタデータ取得）

    alt ① ファイルが存在しない → 404
        TS-->>L: NotFound
        L-->>C: 404（ファイルなし）
    else ② バリデーション NG → 400（サイズ範囲外 / Content-Type 不一致 / マジックナンバー不一致）
        TS-->>L: メタデータ
        L->>TS: DeleteObject（バリデーション前を削除）
        L-->>C: 400（検証失敗・temp 削除済み）
    else ③ 移送後に RDS 保存 失敗 → 400 / 500
        TS-->>L: メタデータ
        L->>PS: CopyObject（バリデーション後S3へ移送）
        L->>P: INSERT
        P->>DB: クエリ実行
        DB-->>P: エラー
        P-->>L: エラー
        L->>PS: DeleteObject（バリデーション後S3のコピーを補償削除）
        L->>TS: DeleteObject（バリデーション前も削除）
        L-->>C: 400 / 500（user_id 不正なら 400、それ以外 500）
    end
```

※ S3 と RDS はまたいだトランザクションにできないため、上図のとおり**失敗時は補償削除**で整合性を保つ（「整合性（S3 と RDS の分散更新）」節を参照）。
